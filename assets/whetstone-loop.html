whetstone — the gate decision tree
gateVerdict() runs after every pass · the model never sees it · first branch that matches wins
CODE OWNS THE GATE
EACH PASS → gateVerdict(state)inputs: score · pass count · best-score history — keep-best already restored the best
1 is the score a finite number 0–100?
a malformed / missing score is not a verdict the model can fake
no → ERROR invalid score, or the pass threw — the run stops
yes ↓
2 score ≥ --target ?
the only branch that can say done — and it lives in code, not a prompt
yes → DONE then --confirm-scorer veto + --stability-runs re-check · the Forge may learn a check
no ↓
3 pass ≥ --cap ? · spend > budget ?
--budget / --budget-tokens are checked after a pass, so may overshoot by one
yes → CAPPED the hard stop — --cap is the true ceiling no unattended run may exceed
no ↓
4 best stalled < --min-delta ?
measured across the last --plateau-window passes of best score
yes → PLATEAU escalation ladder: opus, then Fable 5 — one rung per stall, then it stops
no ↓
↻ RUNNING
score < target, room left → loop back to ACT for one more edit, then gateVerdict runs again
on a DONE, the gate can get STRONGER (--forge)
recovered-veto → learn what the gamed version got wrong
easy-done (1 edit, no confirm) → learn from baseline → final
--gate-self-probe → learn a check for any mutant the gate passed
Precedence  error › done › capped › plateau › running  — the model never runs this decision, it only edits & diagnoses. Only code writes “done”.
